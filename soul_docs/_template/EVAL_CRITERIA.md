## Core Evaluation Dimensions

### 1. Problem Solving Ability

**What they look for:**
- How you approach unfamiliar problems
- Breaking down complex problems
- Asking clarifying questions
- Identifying edge cases

**Scoring indicators:**

**Strong:**
- Immediately asks relevant clarifying questions
- Breaks problem into manageable pieces
- Considers multiple approaches before coding
- Identifies edge cases proactively

**Moderate:**
- Understands problem after discussion
- Can work through solution with guidance
- Catches some edge cases when prompted

**Weak:**
- Struggles to understand problem
- Jumps to coding without planning
- Misses obvious edge cases
- Needs significant guidance

### 2. Technical Competence

**What they look for:**
- Data structure knowledge
- Algorithm familiarity
- Code correctness
- Language proficiency

**Scoring indicators:**

**Strong:**
- Chooses optimal data structures immediately
- Implements algorithms correctly first try
- Handles edge cases in code
- Deep understanding of chosen language

**Moderate:**
- Knows common data structures but may need hints
- Gets algorithm mostly right with minor bugs
- Handles main cases, misses some edges
- Functional knowledge of language

**Weak:**
- Unfamiliar with basic data structures
- Significant algorithmic errors
- Code doesn't handle basic cases
- Struggles with language syntax

### 3. Code Quality

**What they look for:**
- Readability and organization
- Variable naming
- Code structure
- Comments (if needed)

**Scoring indicators:**

**Strong:**
- Clean, readable code
- Meaningful variable names
- Well-structured functions
- Appropriate comments for complex logic
- Follows language conventions

**Moderate:**
- Generally readable
- Decent naming (could be better)
- Reasonable structure
- Some organization issues

**Weak:**
- Hard to read/follow
- Poor naming (single letters, unclear)
- Monolithic functions
- Messy or inconsistent style

### 4. Communication

**What they look for:**
- Explaining thought process
- Asking good questions
- Articulating tradeoffs
- Collaborative problem solving

**Scoring indicators:**

**Strong:**
- Thinks aloud clearly
- Asks insightful questions
- Explains approach before coding
- Discusses tradeoffs explicitly
- Easy to follow and collaborate with

**Moderate:**
- Communicates when prompted
- Answers questions adequately
- Can explain code after writing
- Workable collaboration

**Weak:**
- Silent coding
- Doesn't ask questions
- Can't explain approach clearly
- Difficult to work with

### 5. Analysis & Optimization

**What they look for:**
- Time complexity analysis
- Space complexity analysis
- Identifying optimization opportunities
- Understanding tradeoffs

**Scoring indicators:**

**Strong:**
- Correctly analyzes complexity unprompted
- Identifies optimal solution
- Discusses time/space tradeoffs
- Optimizes without hints

**Moderate:**
- Can analyze complexity when asked
- Finds reasonable solution
- Understands tradeoffs when discussed
- Optimizes with hints

**Weak:**
- Doesn't understand Big-O
- Can't identify inefficiencies
- No consideration of tradeoffs
- Doesn't optimize even with help

### 6. Testing & Debugging

**What they look for:**
- Testing approach
- Edge case identification
- Debugging skills
- Code verification

**Scoring indicators:**

**Strong:**
- Tests with diverse cases
- Identifies edge cases proactively
- Debugs efficiently
- Catches own bugs quickly

**Moderate:**
- Tests basic cases
- Finds some edge cases
- Can debug with hints
- Catches major bugs

**Weak:**
- Doesn't test code
- Misses obvious edge cases
- Struggles to debug
- Doesn't verify correctness
